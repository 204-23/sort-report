# 前言與研究動機  
在電腦科學的領域中，「排序」（Sorting）是最基礎且核心的研究課題之一。無論是數據處理、搜尋優化，還是複雜的系統開發，如何高效地排列資料直接影響了程式的運行效能與資源消耗。本報告針對五種具代表性的排序演算法——氣泡排序（Bubble Sort）、選擇排序（Selection Sort）、插入排序（Insertion Sort）、合併排序（Merge Sort） 以及特殊的 重力排序（Gravity Sort） 進行深入探討。我們不只停留在理論層面的演算法推導，更透過模擬實驗觀察資料量 ($n$) 從 $100$ 增加至 $10,000$ 時，各演算法在執行時間與記憶體空間上的表現差異。透過理論複雜度與實測數據的交叉比對，本報告旨在分析不同排序演算法的「權衡取捨」（Trade-off），探討在面對不同資料特性與硬體限制時，開發者該如何選擇最適當的工具。這不僅是一場性能的競賽，更是一次對於「分治思想」、「物理模型模擬」與「原地運算」等核心邏輯的深度學習紀錄。
# 氣泡排序法 (Bubble Sort)  
原理： 藉由相鄰元素的比較與交換，讓較大的值逐輪「浮」向數列末端。  
操作： 從頭開始兩兩比較，若 $A[i] > A[i+1]$ 則交換，重複 $n-1$ 輪。  

複雜度分析：  
1.時間複雜度： 平均與最差皆為 $O(n^2)$。若加入旗標（Flag）優化，最佳情況（已排序）可達 $O(n)$。  
2.空間複雜度： $O(1)$，僅需常數級額外空間。  
3.穩定性： 穩定排序（Stable）。
# 選擇排序法 (Selection Sort)  
原理： 每輪從未排序區間中挑選「最小值」，並將其置換到已排序區間的末尾。  
操作： 掃描未排序序列找到最小元素，與該序列的第一個位置交換，縮小未排序區間。  

複雜度分析：  
1.時間複雜度： 無論資料初始狀態如何，皆為 $O(n^2)$。  
2.空間複雜度： $O(1)$。  
3.穩定性： 不穩定排序（Unstable），因為交換過程可能改變相同元素的相對順序。
# 插入排序法 (Insertion Sort)  
原理： 將數列分為已排序與未排序，逐一將未排序元素「插入」到已排序序列的正確位置。  
操作： 提取當前元素，與左側已排序元素由右向左比較，直到找到合適位置插入。  

複雜度分析：  
1.時間複雜度： 平均為 $O(n^2)$。在資料幾乎已排序的情況下，效率極高，最佳可達 $O(n)$。  
2.空間複雜度： $O(1)$。  
3.穩定性： 穩定排序（Stable）。
# 合併排序法 (Merge Sort)  
原理： 採用分治法（Divide and Conquer），將大問題拆解為小問題解決後再合併。  
操作： 遞迴地將數列對半拆分至單一元素，再依大小關係兩兩合併。  

複雜度分析：  
1.時間複雜度： 穩定維持在 $O(n \log n)$。  
2.空間複雜度： $O(n)$，合併過程中需要額外的輔助空間。  
3.穩定性： 穩定排序（Stable）。
# 重力排序法 (Gravity/Bead Sort)  
重力排序的空間需求極大，若數列中包含一個極大的數值 $M$，則需要 $O(n \times M)$ 的空間。因此，它在處理數值密集且範圍小的整數時才具有優勢。  
原理： 模擬物理重力現象。將數字轉化為珠子掉落，大的數字珠子多，掉落後會在底部堆疊出長度差異。  
操作： 將正整數轉換為二進制般的格點，模擬珠子在重力作用下下墜至最低位點。  

複雜度分析：  
1.時間複雜度： 在硬體模擬（並行處理）下可達 $O(1)$ 或 $O(\sqrt{n})$，但在軟體模擬中，通常取決於數值的大小（Sum of elements），複雜度與數值的範圍有關。  
2.空間複雜度： 需要較大的空間來模擬珠子掉落的格點。  
3.限制： 通常僅適用於正整數。  

# 綜合比較  
### 效率的「天花板」與「地板」
合併排序是這五者中的「效能王者」。無論資料多亂，它都能穩定在 $O(n \log n)$，在處理萬級以上的數據時，它是唯一具備實用價值的演算法。  
插入排序在特定條件下（資料幾乎已排好）表現驚人，甚至比合併排序更省資源，這是因為它的常數項係數極小。  
氣泡與選擇排序通常只存在於學術討論中，因為它們在處理大規模數據時效率極低。  
### 記憶體的代價 (空間複雜度)  
原地排序 (In-place)： 氣泡、選擇、插入排序都只需要極少的額外空間（$O(1)$），這對記憶體吃緊的微控制器（MCU）非常友善。  
額外空間： 合併排序需要跟原始數據一樣大的輔助空間（$O(n)$），如果記憶體不足，可能會導致程式崩潰。  
重力排序的弱點： 它的空間需求取決於「數值的大小」。如果你要排序 $\{1, 1000000\}$，重力排序會需要龐大的記憶體來模擬珠子掉落，非常不划算。  
### 穩定性 (Stability)  
如果根據「日期」排序訂單，接著想根據「金額」再排一次，穩定的排序（如合併、插入、氣泡）會保留相同金額訂單的原有日期順序；不穩定的排序（如選擇排序）則會打亂原本的日期先後。

### 效能趨勢觀察 
系列： 隨著 $n$ 增加 10 倍，執行時間約增加 100 倍，呈現明顯的拋物線增長。

# 結果呈現與分析  
效率分水嶺： 在 $n < 50$ 時，插入排序的常數項小，表現往往優於合併排序；但當 $n > 1000$ 後，合併排序展現了壓倒性的速度。  
空間與時間的權衡： 合併排序雖然快，但需要 $O(n)$ 的額外空間，在嵌入式或記憶體受限的環境中需謹慎使用。  
重力排序的特殊性： 它是非比較類排序，雖然在處理特定範圍整數時極快，但無法處理負數或浮點數，通用性較低。  

# 心得與結論  
透過這次的實驗與分析，我深刻理解到演算法的選擇並非「越快越好」。  
透過本次實驗的數據模擬，我最深刻的體會是「複雜度理論」與「實際執行時間」之間的精準對應。在 $n=100$ 時，各演算法的差異微乎其微，甚至基礎排序法因其程式邏輯簡單、不需額外的遞迴開銷，表現反而優於合併排序。然而，一旦資料量跨越 $1000$ 的門檻，差距便迅速擴大。尤其是氣泡排序與合併排序在 $n=10000$ 時的對比，前者耗時近 2 秒，後者卻僅需約 10 毫秒，這讓我意識到在開發大規模數據系統時，若選錯演算法，其效能損失將是成百上千倍的代價，這絕非靠硬體升級就能輕易彌補的。  
實驗中也讓我觀察到「空間換取時間」的軟體工程核心原則。合併排序之所以強大，是因為它利用了額外的記憶體空間來暫存子序列，從而大幅降低了比較次數。

反觀重力排序法（Gravity Sort），雖然在概念上突破了傳統「比較式排序」的限制，但在軟體模擬中，其空間複雜度與數值大小（Max Value）掛鉤的特性，使其在面對數值範圍廣大的資料時顯得極其笨重。在評估演算法時，不能只看 CPU 的執行速度，記憶體（RAM）的佔用與硬體架構的適配度同樣是關鍵指標。  
最後，我得出一個結論：演算法的選擇並非盲目追求理論上的最快。  
如果在開發一個低功耗、記憶體僅有幾 KB 的嵌入式系統（如微控制器），且資料量不大，插入排序或許是最佳解答。  
如果是處理像 F1 比賽數據、航空訂票系統等海量資料，合併排序（或其變體 Timsort）絕對是不二之選。  
這次的研究讓我從一個只會寫代碼的學生，轉變為一個能站在效能、成本與資源平衡點上思考的開發者。排序演算法不只是課本上的考題，更是培養我們解決問題嚴謹度的第一步。
